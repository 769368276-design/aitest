# AI执行成功率与稳定性提升计划（分阶段）

## 目标与范围
- 目标：在保持 `browser_use` 为核心引擎前提下，提升页面操作成功率、降低误停与卡死、缩短失败定位时间。
- 范围：执行策略、可观测性、失败复现、策略灰度与安全脱敏。
- 非目标：本计划不切换默认执行引擎，不一次性重写执行器。

## 现状问题归纳
1. 缺少成功率与失败原因的聚合统计，难以判断优化是否真实生效。
2. 动作“无效果”场景闭环偏弱，易出现重复尝试或盲等。
3. 登录/上传等关键流程仍有较多自由决策，环境抖动时稳定性不足。
4. 异常吞噬与敏感信息落盘会影响诊断质量与合规性。

## 分阶段实施步骤

### 阶段A：先补可观测性（必须先做）
1. 定义统一失败原因码（如 preflight/login/assert/http_5xx/no_effect/timeout/manual_stop）。
2. 在执行结束与步骤结束处落盘原因码、首失败步、关键证据摘要。
3. 新增统计入口（管理命令或视图）输出：
   - 总成功率、按项目成功率、按环境成功率；
   - 首失败步分布；
   - 失败原因TopN趋势。
4. 在报告页增加“失败原因码+证据摘要”展示区。

### 阶段B：提升动作成功率（核心）
1. 增强“动作无效果”检测：
   - 统一比较 URL/DOM关键词/关键网络变化；
   - 连续无效果达到阈值后，触发重规划而非继续重复动作。
2. 登录与上传流程状态机化：
   - 识别页面状态 → 执行动作 → 验证结果 → 失败回退；
   - 把关键校验（是否跳离登录页、是否出现上传完成标识）变为显式规则。
3. 优化等待策略：
   - 基于动作类型动态等待（提交类、导航类、输入类分级）；
   - 对慢页面采用递进式短等待与早停组合，降低盲等。

### 阶段C：策略可控与灰度
1. 以 `ai_exec_policy` 为入口补齐可控参数：
   - 无效果阈值、重规划次数、动作重试上限、提交观察窗口。
2. 增加策略模板（保守/均衡/激进）并可按项目或任务切换。
3. 建立A/B对比脚本，按同一批case评估成功率、平均耗时、误停率。

### 阶段D：可靠性与安全收口
1. 收敛静默异常：关键路径改为结构化日志+错误标签，不再纯 `pass`。
2. 统一脱敏：账号密码、token、cookie 不进入 ai_thought/metrics/日志。
3. 增加失败可重放包（最后N步动作+关键网络+DOM摘要）便于复现。

## 具体落地文件（建议）
- 执行与策略核心：`autotest/services/browser_use_runner.py`、`autotest/services/stop_policy.py`
- 队列与路由：`autotest/services/execution_queue.py`
- 配置项：`qa_platform/settings.py`
- 数据与统计：`autotest/models.py`、`autotest/views.py`、`autotest/management/commands/eval_stop_policy.py`
- 报告展示：`autotest/templates/autotest/report_detail.html`
- 回归测试：`autotest/tests.py`

## 验收标准
1. 成功率：核心case集成功率较基线提升（按项目和环境分层可见）。
2. 稳定性：误停率下降，卡死与超时类失败占比下降。
3. 可诊断性：每次失败都能看到明确原因码与证据摘要。
4. 可回退性：所有新增策略均可通过配置关闭或回退到保守模板。
5. 安全性：执行日志与报告不出现明文敏感信息。

## 执行顺序建议
1. 先做阶段A（无数据不优化）。
2. 再做阶段B（直接拉升成功率）。
3. 然后阶段C（可控灰度，避免再次“过激”）。
4. 最后阶段D（稳定运营与合规收口）。
