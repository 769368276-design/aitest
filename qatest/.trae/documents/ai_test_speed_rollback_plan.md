# AI测试执行速度回退计划（保守版）

## 问题现状
- 之前的提速策略过于激进，主要表现为：执行过程过早中断、后续步骤被大量跳过、长页面/慢响应场景误判失败。
- 激进点集中在三类：单步超时强制失败、非阻塞问题自动升级为阻断停止、停止检查触发时机过早。

## 目标
- 将默认行为回退到“稳定优先”。
- 保留可配置能力，支持后续按项目按需重新开启激进策略。
- 确保 browser_use 仍是核心执行引擎。

## 实施步骤
1. 梳理执行链路中的激进策略触发点，明确可配置与不可配置边界。
2. 增加“非阻塞问题是否升级阻断”的显式配置开关，并接入 StopPolicy。
3. 将默认参数调整为保守值：
   - 提高停止检查最小步数阈值；
   - 延长提交后观察窗口；
   - 关闭单步硬超时默认值（保留总超时兜底）。
4. 保持现有覆盖机制（全局 settings + 单次执行 dataset 覆盖）的一致性，避免兼容性回归。
5. 补充/更新单元测试，覆盖：
   - 新开关默认行为；
   - 覆盖参数生效行为；
   - 原有断言失败与步骤归一化逻辑不受影响。
6. 运行相关测试集并确认通过；若有失败，按最小改动修复并复跑。

## 验证标准
- StopPolicy 相关单测通过。
- 断言停止/步骤跳过相关单测通过。
- 无新增诊断错误。
- 执行默认行为从“激进拦截”变为“保守继续执行”。

## 交付说明
- 本次仅回退“默认策略”，不移除原有能力。
- 如需恢复激进模式，可通过配置开关与阈值按环境逐步开启。
