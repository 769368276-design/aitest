# 常见问题处理方案（部署/AI 执行）

这份文档专门针对用户部署后最常见的 3 类问题：
- 无法启动浏览器（AI 测试跑不起来）
- AI 测试截图不显示 / 截图与步骤对不上
- 用例步骤“参考截图”显示裂开

---

## 1) 无法启动浏览器（AI 测试跑不起来）

### 1.1 先做最小自检（最关键）

在项目根目录执行（Windows 用 PowerShell；macOS 用 Terminal）：

```bash
python -m playwright install chromium
python -c "from playwright.sync_api import sync_playwright;p=sync_playwright().start();b=p.chromium.launch(headless=True);b.close();p.stop();print('ok')"
```

能输出 `ok` 说明：浏览器安装和启动都没问题。

### 1.2 常见原因与处理

- **浏览器没装**：执行 `python -m playwright install chromium`。
- **Linux/Docker 缺依赖**：执行 `python -m playwright install-deps chromium`（仅 Linux），或使用项目 Dockerfile 构建镜像。
- **权限/杀软拦截**：把项目目录放到非系统盘或白名单目录，避免被拦截写入浏览器缓存/临时目录。
- **无桌面环境**：必须 headless（系统默认 headless）。不要强行改成有头模式。

## 2) AI 测试截图不显示 / 报告里没图

### 2.1 先判断是不是 media 404

打开某条截图地址（通常是 `/media/...`）：
- 能打开：说明服务端保存和静态托管都 OK，问题可能在前端渲染或权限。
- 404/打不开：说明**生产环境没配置 media 静态服务**。

### 2.2 解决方案

二选一：
- **推荐（生产）**：用 Nginx/网关把 `MEDIA_ROOT` 对应目录映射到 `/media/`。
- **省事（内网演示）**：设置 `DJANGO_SERVE_MEDIA=true` 让 Django 在非 DEBUG 下也托管 media（不建议公网）。

---

## 3) AI 测试“步骤不对/截图和步骤对不上”

常见原因：
- 测试数据不稳定：列表排序、分页内容、弹窗、A/B 灰度导致页面结构变动
- 环境差异：不同分辨率/DPI、不同语言、不同主题导致定位偏差

建议处理：
- 固定测试账号、固定测试数据（最好提前准备“演示账号”）
- 确认项目的 base_url 配置正确
- 对同一用例重复执行 2~3 次对比（看控制台每步截图、Toast、网络证据）
- 复杂页面：建议拆分成多个小用例，减少“一条用例跑全流程”的漂移

---

## 4) 用例步骤里的“参考截图”显示裂开

原因：上传的不是 PNG（比如 JPG/WebP），但页面用 `data:image/png;base64,...` 强制当 PNG 渲染，导致浏览器无法解码。

解决：
- 升级到最新版代码后重新上传参考截图（系统会按真实图片类型渲染）。

## 5) 浏览器能访问内网，但 AI 执行访问不了项目地址

典型现象：
- 用户自己在 Chrome 里能打开内网系统（例如 `http://dev.xxx.local/` 或公司域名）
- 但 AI 执行时在控制台/步骤里提示无法访问、超时、DNS 解析失败、证书错误等

### 5.1 原因说明（不是业务代码逻辑）

AI 执行会启动一个“自动化专用的 Chrome 实例”，它使用独立的浏览器资料目录（profile），通常不会继承用户日常浏览器的：
- 企业代理/VPN 插件登录态
- PAC/代理认证配置
- 企业自签证书/HTTPS 拦截的信任链

所以会出现“用户自己浏览器能访问，但 AI 浏览器访问不了”的情况。

### 5.2 系统已做的自动处理（无需配置项）

当系统检测到“项目地址预检失败”且属于典型内网问题（DNS/代理/证书）时，会自动：
1. 先按原逻辑启动（临时 profile，不影响现有项目）
2. 失败后切换到“持久化专用 profile”再重试
3. 把失败原因分类写进 Step0 提示（DNS/代理/证书/超时/拒绝等）

持久化 profile 会保存在本机临时目录下（按执行人/项目隔离），用于复用企业插件/证书/VPN 登录状态。

### 5.3 客户侧怎么处理（最常见有效）

让客户按下面做一次即可（后续通常不需要重复）：
1. 在 AI 执行触发“切换专用 profile”后，复用该自动化浏览器窗口
2. 在这个窗口里完成企业环境准备（例如：安装/启用企业代理插件、完成 VPN 登录、导入企业证书等）
3. 重新执行用例

### 5.4 常见错误分类与建议

- **域名解析失败（DNS）**：确认已连接企业 VPN、内网 DNS 可用；先在自动化专用浏览器窗口直接打开项目地址验证。
- **代理连接失败/需要代理认证**：确认代理/VPN 已生效；若依赖企业插件代理，确保该插件在自动化专用浏览器窗口已安装并登录。
- **证书不受信任/HTTPS 校验失败**：若为企业自签证书或 HTTPS 拦截，需在自动化专用浏览器窗口导入/信任证书。
- **连接超时/连接被拒绝**：确认目标服务在该机器网络可达且端口开放；检查防火墙/安全软件拦截。

## 6) 录制插件（QA Recorder）怎么安装？录制/回放没反应怎么办？

### 6.1 安装（Chrome）

1. 打开 Chrome 扩展管理页：`chrome://extensions/`
2. 打开右上角“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择项目目录下的：`qa_recorder_extension/`

安装后浏览器右上角会出现 **QA Recorder** 图标（可能在“拼图”菜单里）。

### 6.2 录制的正确打开方式（平台侧）

1. 打开平台录制页：`/autotest/record/`
2. 填写“录制 URL”，点击“开始录制”（平台会新开一个目标页面，并自动附带录制 token）
3. 在新开的目标页面正常操作（点击/输入/选择）
4. 回到录制页看“实时事件”，可“导出 JSON / 保存为用例 / 回放”

### 6.3 常见问题排查

- **点了开始录制，但实时事件一直空**
  - 检查扩展是否已安装并启用（扩展页是否打开）
  - 更新代码后要在 `chrome://extensions/` 对 QA Recorder 点一次“重新加载”
  - 确认新打开的目标页面 URL 带有 `__qa_recorder_token` 和 `__qa_recorder_host` 参数（没有参数扩展不会实时回传）

- **点击“回放”后没有自动操作**
  - 回放需要目标页面处于打开状态：点击回放后应自动打开一个目标页面窗口
  - 若目标页面打开了仍不动：请在目标页面按 `F12` 打开 Console，看是否有报错（通常是元素找不到/跨域限制）
  - 某些系统对自动化事件有拦截（例如安全策略/输入限制），建议先用最短流程验证（打开→点击→输入→保存）
